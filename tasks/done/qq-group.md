# QQ 群机器人回复能力需求草稿

## 一、背景

当前机器人主要面向私聊场景，已经具备普通对话、命令、Todo、天气、RSS、帮助等能力。

现在希望进一步支持 QQ 群聊，但群聊消息频率高、参与人数多，如果机器人对所有消息自动回复，容易出现以下问题：

* 频繁插话，影响正常群聊。
* 将群成员之间的对话误判为对机器人的提问。
* 多人同时聊天时，上下文混乱。
* 回复内容过长，造成刷屏。
* 与其他机器人互相触发，形成循环回复。
* 不同群的使用需求不同，无法统一采用同一种回复策略。

因此，群聊能力需要采用“有限触发、可配置、默认安静”的设计。

---

## 二、目标

为机器人增加 QQ 群聊回复能力，同时保证：

1. 默认不会对群内所有普通消息自动回复。
2. 群成员可以通过明确方式调用机器人。
3. 群管理员可以控制机器人在本群的活跃程度。
4. 群聊与私聊的会话、权限和配置互不干扰。
5. 避免机器人刷屏、误触发和机器人之间循环回复。
6. 现有私聊行为保持不变。

---

## 三、群聊回复模式

每个群可以独立设置回复模式。

### 1. `off`：关闭模式

机器人不处理该群中的普通消息和聊天请求。

是否仍允许管理员执行配置命令，可根据现有权限体系决定。

适合暂时停用机器人，但不希望将机器人移出群聊的场景。

### 2. `command`：仅命令模式

机器人只响应明确命令，例如：

* `/help`
* `/天气 北京`
* `/todo`
* `/ping`

不响应 `@机器人 + 普通问题`，也不主动参与闲聊。

适合工具型群聊或消息较多的群。

### 3. `mention`：提及模式

机器人响应：

* 明确命令。
* `@机器人 + 内容`。
* 用户回复机器人的消息后继续提问。

机器人不响应未明确指向它的普通群聊消息。

该模式建议作为群聊默认模式。

### 4. `active`：主动模式

除命令、@ 和回复消息外，机器人可以根据语义判断主动参与部分群聊。

该模式属于可选增强能力，默认关闭，并需要额外的频率控制和防刷屏机制。

---

## 四、基础触发规则

### 必须响应

在当前群模式允许的前提下，以下消息应进入处理流程：

1. 明确的机器人命令。
2. @机器人并附带有效内容。
3. 回复机器人上一条消息并继续提问。
4. 管理员使用机器人管理命令。

### 默认忽略

以下消息默认不进入 LLM 对话流程：

1. 普通群聊消息。
2. 仅包含表情、图片或无意义短文本的消息。
3. 仅 @机器人但没有实际内容的消息。
4. 其他机器人账号发送的消息。
5. 机器人自己发送的消息。
6. 系统通知、入群通知、撤回通知等非用户聊天事件。
7. 已经处理过的重复事件。

对于“仅 @机器人”的情况，可以选择静默忽略，或者返回一条简短提示，例如：

> 我在，请直接告诉我需要做什么。

该行为应避免频繁触发。

---

## 五、群聊上下文

群聊上下文不能直接复用私聊上下文。

建议至少按以下维度隔离：

* 群 ID。
* 用户 ID。
* 会话类型。
* 当前线程或回复关系。

需要明确群聊采用哪种上下文模型：

### 方案 A：群级共享上下文

同一个群中的成员共享近期聊天上下文。

优点：

* 机器人能够理解群内连续讨论。
* 多人可以围绕同一话题共同交流。

风险：

* 不同成员的话容易互相污染。
* 可能将某人的信息错误关联到另一人。
* 隐私边界较弱。

### 方案 B：群内按用户隔离上下文

同一个群中，每个用户拥有独立会话。

优点：

* 上下文清晰。
* 不容易串话。
* 与私聊模型较接近。

风险：

* 无法自然理解多人共同讨论。
* 回复机器人消息时需要额外关联上下文。

### 初步建议

第一版采用“群 ID + 用户 ID”隔离上下文。

回复机器人消息时，可以附带被回复消息的有限内容，作为本次请求上下文，但不自动读取整个群的全部聊天记录。

后续如果需要多人协作聊天，再单独设计群级会话模式。

---

## 六、回复长度与群聊展示

群聊中的回复应比私聊更克制。

建议：

1. 群聊默认输出较短回复。
2. 超长内容优先总结，必要时提示用户继续追问。
3. Todo、天气、Help 等结构化内容继续支持 Markdown。
4. 避免连续发送大量消息。
5. 一次请求尽量合并成一条或少量消息。
6. 平台存在单条消息长度限制时，沿用现有分段发送能力。
7. 群聊回复不应包含内部用户 ID、群 ID、数据库 ID 或调试信息。

可以考虑为 LLM 增加群聊场景提示：

* 当前处于多人群聊。
* 回复应简洁。
* 只回答明确提问。
* 不主动评价无关群成员。
* 不泄露其他用户的私聊或记忆内容。

---

## 七、防刷屏与冷却机制

群聊需要独立的频率限制。

建议至少支持：

* 单群冷却。
* 单用户冷却。
* 同时请求数量限制。
* 连续回复次数限制。
* 重复消息去重。

初步规则可以是：

1. 同一个群短时间内只允许处理有限数量请求。
2. 同一用户连续快速调用时，进行排队、拒绝或提示稍后再试。
3. `active` 模式主动回复后进入更长冷却。
4. 机器人连续回复达到限制后停止主动参与，直到有新的明确 @ 或命令。
5. 命令和管理员操作可以拥有不同的限流规则。

具体时间和数量不在需求阶段写死，应支持配置，使用合理默认值。

---

## 八、防止机器人互相触发

机器人必须识别消息发送者类型。

至少应做到：

1. 不处理自己发送的消息。
2. 默认不处理其他机器人发送的消息。
3. 不因引用自己的回复而无限重复生成。
4. 对同一平台事件 ID 做幂等处理。
5. 对失败重试产生的重复事件进行去重。

如果平台无法可靠判断对方是否为机器人，应至少通过自身账号 ID、已知机器人列表或事件类型进行过滤。

---

## 九、权限和群级配置

群聊配置应按群独立保存。

可能的配置包括：

* 回复模式。
* 是否允许普通成员使用命令。
* 是否允许主动回复。
* 冷却时间。
* 是否启用 Todo。
* 是否启用天气。
* 是否启用 RSS。
* 是否启用普通 LLM 对话。
* 回复长度偏好。
* 管理员名单或权限来源。

修改群配置时，应限制为：

* 群主。
* 群管理员。
* 机器人全局管理员。
* 项目现有权限系统允许的其他角色。

普通群成员不能随意修改群级配置。

第一版不一定需要一次实现全部配置，可以先实现回复模式和管理员权限。

---

## 十、命令设计草稿

可以增加群聊管理命令，例如：

```text
/group
/group status
/group mode off
/group mode command
/group mode mention
/group mode active
```

中文别名可以后续决定，例如：

```text
/群设置
/群设置 状态
/群设置 模式 提及
```

`/group status` 可以显示：

```text
当前群聊模式：mention
普通命令：已启用
@机器人对话：已启用
主动回复：未启用
```

命令名称应结合项目现有命令风格确定，避免与已有命令冲突。

---

## 十一、群聊与私聊数据边界

必须保证：

1. 群聊中不能自动读取或暴露用户私聊历史。
2. 私聊记忆不能默认在群聊中直接使用。
3. 用户个人 Todo 不应因群内其他成员执行命令而暴露。
4. 群 Todo 与个人 Todo 如果同时存在，需要明确区分。
5. 群聊生成的记忆不能默认进入个人长期记忆。
6. 群内任何用户不能通过提示词获取其他用户数据。

第一版建议：

* 普通对话按“群 ID + 用户 ID”隔离。
* `/todo` 默认仍操作调用者自己的 Todo，且输出前评估是否适合在群内公开。
* 涉及个人敏感数据的命令，可以拒绝在群聊展示，并提示用户私聊机器人。
* 长期记忆功能在群聊中默认关闭，除非后续单独设计群记忆。

---

## 十二、异常处理

群聊回复失败时：

1. 不应连续重试并刷屏。
2. 不应输出完整内部错误栈。
3. 可以返回简短失败提示。
4. 超时后应取消或忽略过期结果。
5. 如果用户后续又发起了新请求，旧请求的迟到结果不应干扰新对话。
6. LLM 不可用时，命令类功能应尽可能独立运行。
7. 管理配置失败时，应明确提示未生效，不能伪造成功状态。

---

## 十三、第一阶段建议范围

第一阶段只实现稳定、可控的基础群聊能力：

### 必须实现

1. 识别群聊消息和私聊消息。
2. 群聊默认采用 `mention` 模式。
3. 支持命令触发。
4. 支持 @机器人触发。
5. 支持回复机器人消息继续对话。
6. 忽略机器人自身和其他机器人消息。
7. 群聊会话按“群 ID + 用户 ID”隔离。
8. 增加基础群级回复模式配置。
9. 只有管理员可以修改群配置。
10. 增加基础限流和重复事件去重。
11. 保持现有私聊行为不变。
12. 补充群聊相关日志、测试和说明文档。

当前实现采用“默认 `mention`、有限触发”的策略；仍可通过
`QQ_MAID_GROUP_MESSAGE_MODE=off` 显式关闭普通群消息。

### 暂不实现

1. 基于语义的主动插话。
2. 读取整个群的历史聊天记录。
3. 多人共享的群级长期上下文。
4. 群级长期记忆。
5. 复杂的群 Todo 权限体系。
6. Web 管理后台。
7. 每个群高度细粒度的功能开关。

---

## 十四、后续可选能力

基础版本稳定后，可以继续增加：

* `active` 主动参与模式。
* 群级公共 Todo。
* 群级公共记忆。
* 群聊每日摘要。
* RSS 推送到指定群。
* 定时 Todo 提醒到指定群。
* 新成员欢迎语。
* 群内问答知识库。
* 群消息关键词触发。
* 不同群使用不同角色提示词。
* 群级模型和回复长度配置。
* 安静时段。
* 白名单和黑名单。
* 管理员审核后开启主动回复。

---

## 十五、验收方向

基础群聊版本完成后，应满足：

1. 私聊功能和现有行为不受影响。
2. 机器人加入群后，不会回复所有普通聊天内容。
3. 默认情况下，用户通过命令或 @机器人可以正常获得回复。
4. 用户回复机器人的消息时可以继续对话。
5. 不同群、不同用户的上下文不会串联。
6. 其他机器人消息不会触发回复。
7. 同一事件重复投递不会产生重复回复。
8. 普通成员不能修改群级配置。
9. 机器人高频被调用时不会无限刷屏。
10. 群聊中不会泄露其他用户的私聊、Todo、记忆或内部标识。
11. 群聊回复失败时只产生有限、明确的错误提示。
12. 相关测试、构建和静态检查能够通过。
