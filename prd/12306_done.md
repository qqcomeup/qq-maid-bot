# 任务：接入 12306 列车时刻查询，并支持火车行程 Todo

## 背景

项目现有 Todo 和 LLM 自然语言解析能力。

现已确认以下 12306 接口可以正常返回指定日期、指定车次的完整经停时刻：

```text
POST https://mobile.12306.cn/wxxcx/wechat/main/travelServiceQrcodeTrainInfo
Content-Type: application/x-www-form-urlencoded

trainCode=G1&startDay=20260624
```

本次需求原始目标包含两个功能：

1. `/火车 Gxxxx`：查询列车时刻信息。
2. `/todo add ...`：识别火车行程，查询真实到发时间后创建 Todo。

当前阶段两个功能均已落地：`/火车` 查询和 `/todo add` 火车行程识别。

本次只做列车时刻和行程提醒，不做余票、登录、购票或抢票。

## 功能一：列车信息查询

支持命令：

```text
/火车 G1
/火车 G1 明天
/火车 G1 2026-06-28
```

规则：

1. 解析车次和日期。
2. 未提供日期时默认查询今天。
3. 调用 12306 接口查询完整时刻表。
4. 展示：

   * 车次；
   * 查询日期；
   * 始发站和终到站；
   * 是否在指定日期开行；
   * 各经停站的到达时间、出发时间、停车时间；
   * 跨日信息。
5. 指定日期没有数据时，明确提示“该日期未查询到开行信息”，不要擅自改查其他日期并冒充当天结果。

建议输出：

```md
## 🚄 G1 列车时刻

**日期：** 2026-06-24  
**行程：** 北京南 → 上海

| 站序 | 车站 | 到达 | 出发 | 停留 |
|---:|---|---:|---:|---:|
| 1 | 北京南 | --:-- | 07:00 | -- |
| 2 | 南京南 | 10:20 | 10:22 | 2 分钟 |
| 3 | 上海 | 11:32 | --:-- | -- |

> 时刻信息来自铁路查询服务，请以车站实际公告为准。
```

同时提供可独立阅读的纯文本输出。

## 功能二：火车行程 Todo

支持结构化输入：

```text
/todo add G34 杭州东 北京南 明天 05车12A 8站台
```

也支持自然语言：

```text
/todo add 明天坐 G34 从杭州东去北京南，座位 05 车 12A，8 站台
```

座位号和站台为可选信息。

### LLM 解析

可以复用现有 Todo LLM 流程，将用户输入解析成结构化结果：

```json
{
  "kind": "train",
  "train_code": "G34",
  "from_station": "杭州东",
  "to_station": "北京南",
  "travel_date": "2026-06-24",
  "seat": "05车12A",
  "platform": "8站台",
  "note": null
}
```

要求：

1. 优先复用现有 Todo 模型配置和结构化输出能力。
2. LLM 只负责理解输入，不负责生成列车时刻。
3. 车次、站点和时间必须通过 12306 返回结果校验。
4. LLM 不得猜测发车时间、到达时间、座位号或站台。
5. 缺少车次、出发站、到达站或日期时，返回补充提示，不创建 Todo。
6. 普通 Todo 输入不得被错误识别为火车行程。

### 时刻查询与校验

解析完成后：

1. 使用 `travel_date + train_code` 查询完整时刻表。
2. 在经停站中找到出发站和到达站。
3. 校验出发站位于到达站之前。
4. 读取：

   * 出发站的 `startTime`；
   * 到达站的 `arriveTime`；
   * 对应 `dayDifference`。
5. 按中国标准时间生成完整的出发和到达日期时间。
6. Todo 的时间使用出发站的实际发车时间。
7. 跨日到达必须正确增加日期。

站名可以做最小归一化，例如忽略末尾的“站”，但不要把“杭州”静默替换成“杭州东”。存在歧义时应提示用户输入具体车站。

### Todo 内容

查询成功后，复用现有 Todo 创建、存储、编号和提醒机制。

建议标题：

```text
🚄 G34 杭州东 → 北京南
```

建议详情：

```text
乘车日期：2026-06-24
出发：07:05
到达：11:40
座位：05车12A
站台：8站台
```

座位号和站台为空时不显示对应字段。

创建成功输出示例：

```md
## 🚄 已添加火车行程

- **车次：** G34
- **行程：** 杭州东 → 北京南
- **出发：** 2026-06-24 07:05
- **到达：** 2026-06-24 11:40
- **座位：** 05车12A
- **站台：** 8站台

已按现有 Todo 提醒规则保存。
```

不要为了火车功能复制一套新的提醒系统。

## 12306 Provider

将接口封装成独立、可替换的 Provider，不要把 HTTP 调用直接写进 Todo Flow。

建议能力：

```rust
async fn query_train_schedule(
    train_code: &str,
    travel_date: NaiveDate,
) -> Result<TrainSchedule>;
```

建议数据结构：

```rust
struct TrainSchedule {
    train_code: String,
    travel_date: NaiveDate,
    start_station: String,
    end_station: String,
    stops: Vec<TrainStop>,
}

struct TrainStop {
    station_name: String,
    arrive_time: Option<String>,
    departure_time: Option<String>,
    stopover_minutes: Option<u32>,
    day_difference: i32,
    station_train_code: String,
}
```

接口请求：

```text
POST https://mobile.12306.cn/wxxcx/wechat/main/travelServiceQrcodeTrainInfo
```

表单字段：

```text
trainCode=<大写车次>
startDay=<YYYYMMDD>
```

从返回值中读取：

```text
data.trainDetail.stopTime
```

重点字段包括：

```text
stationName
arriveTime
startTime
stopover_time
dayDifference
stationTrainCode
```

实现时以实际接口响应为准，不能假定所有字段永远存在。

## 稳定性要求

1. 设置合理的连接和请求超时。
2. 检查 HTTP 状态码和业务响应结构。
3. 区分：

   * 日期无开行数据；
   * 车次格式错误；
   * 站点不匹配；
   * 接口超时；
   * 响应结构变化。
4. 不要将接口异常解释成“车次不存在”。
5. 可以对同一日期、同一车次的结果做短期缓存，例如 10 分钟。
6. 不需要登录、Cookie、账号或验证码。
7. 日志中不要记录敏感配置或完整原始响应。
8. 该接口不是正式开放 API，代码结构必须允许以后替换数据源。

## 错误提示

缺少日期：

```text
请补充乘车日期，例如：
/todo add G34 杭州东 北京南 明天
```

站点不匹配：

```text
查询到 G34 的时刻表，但未找到“杭州”站。
请使用具体车站名称，例如“杭州东”。
```

站点顺序错误：

```text
G34 的运行方向中，北京南位于杭州东之前，请检查出发站和到达站。
```

接口不可用：

```text
铁路时刻服务暂时不可用，本次没有创建 Todo，请稍后重试。
```

## 验收标准

1. `/火车 G1 2026-06-24` 能展示该日期的完整经停时刻。
2. `/火车 G1` 默认查询当天。
3. `/todo add G34 杭州东 北京南 明天` 能查询时刻并创建 Todo。
4. 自然语言形式能够由 LLM 正确整理。
5. 座位号和站台能够保存并展示，但不参与 12306 查询。
6. Todo 时间等于指定出发站的实际发车时间。
7. 跨日到达日期计算正确。
8. 出发站或到达站不匹配时不创建 Todo。
9. 接口超时或响应异常时不创建错误 Todo。
10. 普通 `/todo add 买牛奶` 等行为保持不变。
11. Markdown 和纯文本输出均可独立阅读。
12. 旧 Todo 数据、用户侧编号和提醒逻辑不受影响。
