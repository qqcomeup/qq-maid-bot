# 任务：为长期记忆增加个人/群聊作用域并修复跨用户记忆泄露

## 背景

当前项目的长期记忆存在严重的作用域隔离问题。

已观察到的现象：

* 用户 A 在群聊中执行 `/记忆`，可以看到一组长期记忆；
* 同一个群内的用户 B 执行 `/记忆`，也能看到完全相同的记忆；
* 这说明至少记忆列表或管理链路没有按当前用户隔离；
* 目前还需要继续排查普通聊天中的记忆检索、Prompt 注入、缓存和 Pending 操作是否也存在跨用户或跨群读取问题。

本次任务不是只修复 `/记忆` 的展示结果，而是要建立明确、统一、可验证的记忆作用域模型，并在所有记忆读写链路中严格执行。

## 任务类型

* 严重 Bug 修复
* 记忆权限与作用域模型调整
* 数据结构与兼容性改造
* 命令行为扩展

## 核心目标

长期记忆分为两种相互独立的作用域：

1. **个人记忆**

   * 归属于具体用户；
   * 跟随用户跨私聊和不同群聊使用；
   * 其他用户无权查看、修改、删除或注入聊天上下文。

2. **群记忆**

   * 归属于具体群聊；
   * 只在该群内有效；
   * 当前群成员可以在该群聊天时使用；
   * 不能在私聊或其他群中使用。

修改完成后，任何记忆查询、管理、检索、缓存、Prompt 注入和 Pending 操作都不得跨用户或跨群泄露。

## 一、目标作用域模型

### 1. 个人记忆

逻辑归属：

```text
scope_type = personal
scope_id = 当前用户的稳定 user_id
```

规则：

* 当前用户只能查看自己的个人记忆；
* 当前用户只能修改、删除自己的个人记忆；
* 其他用户不能看到这部分记忆；
* 用户私聊机器人时可以使用；
* 用户在任意群聊发言时也可以使用；
* 个人记忆跟随用户，不跟随会话或群聊。

### 2. 群记忆

逻辑归属：

```text
scope_type = group
scope_id = 当前群的稳定 group_id
```

规则：

* 只在对应群聊中使用；
* 不能在私聊中使用；
* 不能在其他群聊中使用；
* 不得因为同一个用户同时位于多个群，就把某个群的记忆带到另一个群；
* 群记忆应保存创建者身份，供编辑和删除时做权限判断。

建议字段：

```text
scope_type
scope_id
created_by_user_id
```

具体字段名、类型和表结构以仓库现状为准，但语义必须明确，不能继续使用含义模糊、可混用的单一 owner/scope 字段来隐式区分。

## 二、聊天时的记忆读取规则

### 1. 私聊场景

当前用户 `U` 私聊机器人时：

```text
允许读取：
- personal:U

禁止读取：
- 任意群记忆
- 其他用户的个人记忆
- 无法确认归属的旧记忆
```

最终记忆上下文：

```text
当前用户的个人记忆
```

### 2. 群聊场景

当前用户 `U` 在群 `G` 中发消息时：

```text
允许读取：
- personal:U
- group:G

禁止读取：
- 其他用户的个人记忆
- 其他群的群记忆
- 无法确认归属的旧记忆
```

最终记忆上下文：

```text
当前发言人的个人记忆
+
当前群的群记忆
```

### 3. 无个人记忆时

若当前群成员没有个人记忆，但当前群存在群记忆：

```text
只使用当前群记忆
```

不得回退到其他用户的个人记忆、全局记忆或最近一次命中的记忆。

## 三、命令行为

现有中英文命令别名应继续兼容，以下使用 `/记忆` 表示对应的记忆命令。

### 1. 个人记忆命令

无论在私聊还是群聊，以下命令始终操作当前用户的个人记忆：

```text
/记忆
/记忆 list
/记忆 show <编号>
/记忆 add <内容>
/记忆 edit <编号> <新内容>
/记忆 delete <编号>
```

行为要求：

* `/记忆` 默认只显示当前操作用户的个人记忆；
* 用户在群里执行 `/记忆`，不能看到其他群成员的个人记忆；
* 个人记忆新增时绑定当前用户；
* show/edit/delete 都必须再次校验该记录属于当前用户；
* 即使用户提供真实数据库 ID，也不能绕过作用域校验。

### 2. 群记忆命令

新增显式的群作用域命令：

```text
/记忆 群
/记忆 群 list
/记忆 群 show <编号>
/记忆 群 add <内容>
/记忆 群 edit <编号> <新内容>
/记忆 群 delete <编号>
```

如项目同时支持英文命令，应提供等价语法，例如：

```text
/memory group
/memory group add <内容>
/memory group show <编号>
/memory group edit <编号> <新内容>
/memory group delete <编号>
```

具体别名风格应与现有命令解析方式保持一致，不要单独实现一套不兼容的解析器。

群记忆行为：

* 只能在群聊中操作；
* `/记忆 群` 只显示当前群的群记忆；
* 新增群记忆时绑定当前群；
* 同一个群的成员可以在聊天时使用这些记忆；
* 不得访问其他群的群记忆。

私聊中执行群记忆命令时，返回明确提示，例如：

```text
群记忆只能在群聊中查看或管理。
```

不得猜测用户最近所在群，也不得选择任意默认群。

## 四、群记忆管理权限

群记忆至少要保存：

```text
created_by_user_id
```

第一版建议规则：

* 当前群成员可以查看当前群的群记忆；
* 当前群成员可以新增群记忆；
* 普通成员只能编辑、删除自己创建的群记忆；
* 如果项目已有可靠的群管理员、机器人管理员或超级用户识别机制，可以允许管理员管理当前群全部群记忆；
* 如果当前无法可靠识别管理员，不要伪造权限能力，先采用“创建者管理自己记录”的规则。

禁止默认实现为“群内任意成员都能删除或编辑全部群记忆”。

权限校验失败时，不要泄露目标记录实际属于谁或属于哪个群，返回通用提示即可。

## 五、必须审计和修复的完整链路

不要只修改 `/记忆` 列表。请逐项检查以下链路。

### 1. 数据库存储

检查当前记忆表、migration 和存储接口：

* 当前记忆是按什么字段归属；
* 是否只使用了 `scope_key`、`owner_key` 或类似模糊字段；
* 是否存在群成员共享同一个 owner 的情况；
* 是否需要新增明确的 `scope_type`、`scope_id`、`created_by_user_id`；
* 所有查询、更新、删除是否都带有作用域条件；
* 是否存在仅凭 memory ID 操作记录的接口。

要求所有更新和删除在 SQL 或存储层同时校验目标作用域，不能只依赖上层提前检查。

错误示例：

```sql
DELETE FROM memories WHERE id = ?
```

目标语义应类似：

```sql
DELETE FROM memories
WHERE id = ?
  AND scope_type = ?
  AND scope_id = ?
```

群记忆还要结合创建者或管理员权限检查。

### 2. 记忆写入

排查：

* 私聊手动新增记忆；
* 群聊中普通 `/记忆 add`；
* 群聊中 `/记忆 群 add`；
* LLM 自动提取或自动保存记忆；
* Pending 确认后写入；
* 编辑后重新保存；
* 兼容旧命令时的默认作用域。

要求：

* 普通 `/记忆 add` 永远写入当前用户个人作用域；
* `/记忆 群 add` 才写入当前群作用域；
* 自动记忆默认只能写个人记忆；
* 不允许模型根据聊天内容自行把普通消息升级为群公共记忆；
* 群记忆应要求用户显式使用群作用域命令。

### 3. 记忆列表和管理

排查：

* list；
* show；
* edit；
* delete；
* 编号映射；
* 候选选择；
* 批量操作；
* 中英文别名；
* Pending 确认。

要求：

* 个人编号只能在当前用户个人记忆列表中解析；
* 群编号只能在当前群记忆列表中解析；
* 不允许个人和群使用同一份未隔离的临时列表；
* 不允许通过内部 ID 跨作用域操作；
* 查询不到时不得扩大搜索范围。

### 4. 普通聊天记忆检索

排查普通聊天处理链路中：

* 何时加载长期记忆；
* 使用什么用户、会话或 scope 参数；
* 检索函数是否查询全部记忆；
* 是否先全局 top-k，再在应用层过滤；
* 是否在缺少 user_id 时回退到 group、session 或全局；
* 是否把群内某个成员的记忆作为整个群会话记忆。

要求：

* 必须先限定合法作用域，再做关键词、FTS、向量或其他检索；
* 不能先全局召回 top-k 再过滤；
* 群聊中应分别检索：

  * 当前用户个人记忆；
  * 当前群群记忆；
* 然后合并结果；
* 合并时保留作用域信息，避免后续缓存或 Prompt 处理丢失来源。

### 5. Prompt 注入

排查：

* `memory_context` 的构造位置；
* 记忆内容进入 system/user message 的位置；
* 群聊上下文是否会被整个会话复用；
* 会话压缩、摘要或 session context 是否把某个用户的个人记忆永久写入群公共上下文。

要求：

* 只允许注入当前请求可访问的记忆；
* 群聊中个人记忆只服务于当前发言者；
* 不得把当前用户个人记忆持久化进群公共摘要；
* 下一位群成员发言时必须重新按其 user_id 构造记忆上下文；
* 私聊不得注入群记忆。

同时在 Prompt 中增加隐私约束，语义类似：

```text
个人记忆仅用于理解当前发言用户和个性化回复。
在公开群聊中，不要主动披露、复述或列举个人隐私记忆；
除非记忆所有者在当前消息中明确要求讨论该内容。
```

不要硬编码具体用户或具体示例内容。

### 6. 缓存

排查任何记忆缓存、上下文缓存、检索缓存：

错误缓存键示例：

```text
group_id
session_id
scope_key
```

如果缓存内容包含“当前用户个人记忆 + 当前群记忆”，缓存键至少必须能区分：

```text
conversation_type
user_id
group_id
记忆版本或更新时间
```

也可以将个人记忆和群记忆分别缓存后再合并。

要求：

* 用户 A 在群里的记忆结果不得被用户 B 复用；
* 群 A 的结果不得被群 B 复用；
* 记忆新增、编辑、删除后，应正确失效相关缓存；
* 不要为了简单直接关闭所有缓存，除非仓库现状证明缓存无法安全修复，并在完成报告中说明。

### 7. Pending 操作

所有记忆相关 Pending 应保存并校验：

```text
initiator_user_id
target_scope_type
target_scope_id
target_memory_id 或候选快照
```

后续确认、修改或选择时：

* 当前用户必须与发起人一致；
* 当前会话必须仍有权操作原目标作用域；
* 不能由群内其他成员接管；
* 不能在创建 Pending 时是个人作用域，确认时变成群作用域；
* Pending 中不得仅保存用户可见编号而不保存已解析的真实目标和作用域。

如果项目已有统一 Pending 发起人保护逻辑，应复用，不要重复实现另一套。

### 8. 会话摘要和压缩

检查 compact、session context、历史摘要等链路：

* 是否把个人记忆或个人化信息写入群级持久摘要；
* 是否按群共享摘要；
* 是否在下一名用户发言时继续使用上一名用户的个人化上下文。

要求：

* 长期记忆和会话摘要概念上保持分离；
* 当前用户个人记忆不能被固化到群公共摘要；
* 如群聊摘要不可避免包含对话事实，应确保它不是作为“某个成员的个人长期记忆”被其他成员管理或查询；
* 不要借本任务大规模重构摘要系统，但必须修复直接导致记忆泄露的耦合。

## 六、严格隔离与失败策略

记忆属于敏感数据，必须采用 fail-closed 策略。

### 禁止作用域回退

禁止以下行为：

* 缺少 user_id 时查询全部个人记忆；
* 查不到个人记忆时改查群记忆；
* 查不到当前群记忆时改查其他群；
* 缺少 group_id 时使用最近群、默认群或 scope_key；
* 作用域解析失败时返回全局结果；
* 为兼容旧数据而把未知归属记忆暴露给所有人。

正确行为：

```text
缺少可靠 user_id：
- 不读取、不管理个人记忆

缺少可靠 group_id：
- 不读取、不管理群记忆

作用域信息无法确认：
- 拒绝操作或返回空结果
```

宁可少返回，也不能扩大访问范围。

## 七、旧数据迁移与兼容

检查现有数据库中的长期记忆是否具备可靠归属信息。

迁移原则：

1. 能根据现有字段明确判断用户归属的记录：

   * 迁移为个人记忆。

2. 能明确判断群归属的记录：

   * 迁移为群记忆。

3. 无法明确判断归属的记录：

   * 不得默认公开给某个群或所有用户；
   * 建议标记为 `legacy/unassigned`、禁用，或放入管理员可处理的隔离状态；
   * 这类记录默认不参与聊天检索；
   * 普通用户不能通过 `/记忆` 查看；
   * 在完成报告中说明数量和处理方式。

如果仓库当前没有适合的未归属状态，也可以采用更小改动的安全方案，例如：

* 将无法确认归属的旧记录设为不可用；
* 或 migration 不自动迁移，并提供明确日志。

但绝不能将全部旧记忆默认归到当前群、首个用户或全局作用域。

Migration 必须：

* 可重复执行或遵循现有 migration 机制；
* 不破坏现有数据库启动；
* 对已有正确归属的数据保持兼容；
* 对异常数据给出日志或统计，不输出原始敏感记忆内容。

## 十一、验收标准

### 1. 个人记忆管理隔离

准备两个不同用户 A、B。

在同一个群内：

1. A 新增一条个人记忆；
2. A 执行 `/记忆` 可以看到该记录；
3. B 执行 `/记忆` 看不到该记录；
4. B 不能通过编号或内部 ID 查看、编辑、删除该记录；
5. B 的个人记忆也不能被 A 查看或管理。

### 2. 群记忆管理隔离

准备群 G1、G2：

1. 在 G1 新增一条群记忆；
2. G1 中执行 `/记忆 群` 可以看到；
3. G2 中执行 `/记忆 群` 看不到；
4. 私聊执行 `/记忆 群` 被明确拒绝；
5. G2 不能通过编号或内部 ID 操作 G1 的群记忆。

### 3. 私聊聊天读取

用户 A 私聊机器人时：

* 可以使用 A 的个人记忆；
* 不能使用其他用户个人记忆；
* 不能使用任何群记忆。

### 4. 群聊聊天读取

用户 A 在群 G 发言时：

* 可以使用 A 的个人记忆；
* 可以使用 G 的群记忆；
* 不能使用用户 B 的个人记忆；
* 不能使用其他群记忆。

用户 B 在同一群发言时：

* 可以使用 B 的个人记忆；
* 可以使用 G 的群记忆；
* 不能使用 A 的个人记忆。

### 5. 无记忆回退

* 当前用户没有个人记忆时，只返回当前群合法的群记忆；
* 当前群没有群记忆时，只返回当前用户个人记忆；
* 两者都没有时，记忆上下文为空；
* 不得返回其他作用域内容。

### 6. Pending 隔离

* A 发起记忆编辑或删除确认后，B 不能继续或确认；
* Pending 作用域不能在后续消息中变化；
* 无权限的确认不会删除或修改任何记录。

### 7. 缓存隔离

* A 在群中发言后，B 紧接着发言，不会继承 A 的个人记忆；
* 不同群不会共享群记忆缓存；
* 新增、编辑、删除后，相关检索结果能正确刷新。

### 8. 旧数据安全

* 无法确认归属的旧记忆不会出现在普通用户列表中；
* 不会注入普通聊天；
* migration 不会把未知记录公开给全部用户或群。

### 9. 兼容性

* 原有个人 `/记忆`、`/memory` 命令保持可用；
* 原有个人记忆在能可靠确认归属时继续可用；
* 普通聊天、命令渲染和现有数据库启动不受无关影响。

建议按以下顺序处理：

## 十四、完成后输出

完成后请提供报告，包含：

   * 为什么不同群成员会看到同一批个人记忆；

   * 新增或调整了哪些字段；
   * 旧数据如何分类；
   * 无法确认归属的数据如何处理；
   * 是否存在需要人工处理的数据。

   * 尚未解决的风险；
   * 无法验证的链路；
   * 后续建议。
